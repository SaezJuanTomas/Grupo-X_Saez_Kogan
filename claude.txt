================================================================================
1. DESCRIPCION GENERAL
================================================================================

Aplicacion web full-stack para la gestion de vulnerabilidades de seguridad
informatica. El sistema permite a equipos de seguridad registrar, clasificar,
asignar, dar seguimiento y reportar vulnerabilidades en empresas clientes.

Casos de uso principales:
  - Admin: CRUD completo de vulnerabilidades, empresas, usuarios;
           dashboard de estadisticas; asignacion a analistas
  - Analyst: visualizar sus vulnerabilidades asignadas, agregar comentarios
  - Sistema: refresh tokens con rotacion, historial de cambios automatico,
             rate limiting, integracion con n8n para NVD


================================================================================
2. ARQUITECTURA
================================================================================

Frontend (React + Vite) ----HTTP----> Backend (FastAPI) ----SQL----> PostgreSQL

  - Frontend: http://localhost:5173
  - Backend:  http://localhost:8000
  - Backend API: http://localhost:8000/docs (Swagger UI)

Patron de capas en backend:
  Router (HTTP) -> Service (logica de negocio) -> Repository (base de datos)

Flujo tipico de request:
  Request HTTP
    -> LoggingMiddleware (request ID + timing)
    -> SlowAPIMiddleware (rate limiting)
    -> CORSMiddleware
    -> Exception Handlers (errores globales)
    -> Router -> Service -> Repository -> SQLAlchemy -> DB

6. SISTEMA DE AUTENTICACION
================================================================================

6.1 JWT Access Token
  - Algoritmo: HS256
  - Payload: { sub: user_id, role: str, type: "access", ver: token_version, exp: timestamp }
  - Duracion: 30 minutos (configurable via ACCESS_TOKEN_EXPIRE_MINUTES)
  - Incluye "ver" (token_version) para permitir revocacion

6.2 Refresh Token
  - Payload: { sub: user_id, jti: UUID, type: "refresh", exp: timestamp }
  - Duracion: 7 dias (configurable via REFRESH_TOKEN_EXPIRE_DAYS)
  - Almacenado en tabla refresh_tokens con jti unico
  - Rotacion: cada uso invalida el anterior (revoked=True)
  - Reuso detectado: si se usa un refresh token ya revocado, se rechaza

6.3 Flujo de login
  1. POST /auth/login con { username, password }
  2. Verificar password con bcrypt
  3. Verificar que el usuario este activo
  4. Generar access_token (JWT con token_version actual)
  5. Generar refresh_token (JWT + registro en DB)
  6. Devolver ambos tokens + datos del usuario

6.4 Flujo de refresh
  1. POST /auth/refresh con { refresh_token }
  2. Decodificar JWT, verificar tipo "refresh"
  3. Buscar RefreshToken por jti, que no este revocado y no haya expirado
  4. Marcar como revoked=True
  5. Generar NUEVO par access_token + refresh_token
  6. Devolver nuevos tokens

6.5 Flujo de logout
  1. POST /auth/logout con Bearer token
  2. Revocar todos los refresh_token activos del usuario (revoked=True)
  3. Incrementar token_version del usuario (invalida todos sus access tokens previos)

6.6 Verificacion de acceso (get_current_user)
  1. Extraer Bearer token del header Authorization
  2. Decodificar JWT, verificar tipo "access"
  3. Obtener user_id del payload
  4. Cargar usuario de DB
  5. Verificar que el usuario exista y este activo
  6. Verificar que token_version del JWT coincida con el del usuario
     (si no coincide, el token fue emitido antes de un logout y esta revocado)

6.7 Jerarquia de permisos
  - get_current_user: cualquier usuario autenticado (incluyendo viewer)
  - require_admin: solo rol "admin"
  - require_analyst: roles "admin" o "analyst"

7.4 FLUJO TIPICO (ej: crear vulnerabilidad)
  POST /vulnerabilidades (admin)
    => vulnerability_router.create_vulnerability()
      => require_admin (verifica token + rol admin)
      => VulnerabilityService.create_vulnerability(...)
        => VulnerabilityRepository.create(...)  [INSERT en DB]
        => HistoryRepository.create(...)        [INSERT en history_logs]
        => flush()
      => return VulnerabilityRead

7.5 FLUJO DE ACTUALIZACION CON AUDITORIA (ej: cambiar estado)
  PATCH /vulnerabilidades/1 { "status": "Resuelto" }
    => vulnerability_router.update_vulnerability()
      => get_current_user()  [verifica token]
      => verifica si cambia status o assignee y si es admin
      => VulnerabilityService.update_vulnerability(1, changes, actor_id=user.id)
        => VulnerabilityRepository.get_by_id(1)
        => guarda previous_status y previous_assignee
        => VulnerabilityRepository.update(vuln, status="Resuelto")
        => si cambia status: HistoryRepository.create(action="Estado cambiado", ...)
        => si cambia assignee: HistoryRepository.create(action="Asignado", ...)
        => flush()
      => return VulnerabilityRead

10. FRONTEND
================================================================================

Stack: React 19 + TypeScript + Vite + React Router v6 + Axios + Tailwind CSS

API Client (lib/api.ts):
  - Axios instance con baseURL desde VITE_API_URL (default http://localhost:8000)
  - Interceptor request: agrega Bearer token desde localStorage
  - Interceptor response: captura 401, hace refresh automatico:
      1. Guarda refresh_token de localStorage
      2. POST /auth/refresh
      3. Actualiza access_token y refresh_token en localStorage
      4. Reintenta la request original
      5. Si hay multiples requests en cola, todas se resuelven con el nuevo token
      6. Si falla el refresh, limpia localStorage y redirige a /login

  Funciones exportadas:
    login(payload)         -> POST /auth/login
    logout()               -> POST /auth/logout
    getUsers()             -> GET /usuarios
    getCompanies()         -> GET /empresas
    getCompany(id)         -> GET /empresas/{id}
    createCompany(payload) -> POST /empresas
    updateCompany(id, ...) -> PATCH /empresas/{id}
    getVulnerabilities()   -> GET /vulnerabilidades?page=1&page_size=200
    getVulnerability(id)   -> GET /vulnerabilidades/{id}
    createVulnerability()  -> POST /vulnerabilidades
    updateVulnerability()  -> PATCH /vulnerabilidades/{id}
    deleteVulnerability()  -> DELETE /vulnerabilidades/{id}
    getComments(vulnId)    -> GET /comentarios?vulnerability_id=...
    createComment()        -> POST /vulnerabilidades/{id}/comentarios
    getHistory(vulnId)     -> GET /historial?vulnerability_id=...
    getStats()             -> GET /estadisticas
    getTrends(days)        -> GET /estadisticas/tendencias?days=...
    createUser(payload)    -> POST /usuarios
    updateUser(id, ...)    -> PATCH /usuarios/{id}

AuthContext (context/AuthContext.tsx):
  - Provider que maneja sesion del usuario
  - login(): llama a apiLogin, guarda access_token, refresh_token, session en localStorage
  - logout(): llama a apiLogout, limpia localStorage
  - loadSession(): recupera sesion de localStorage al recargar pagina
  - user: SessionUser | null (id, username, role)
  - error: string | null

App.tsx (componente principal):
  - Punto de entrada de la aplicacion
  - Carga inicial de datos (usuarios, empresas, vulnerabilidades, stats) via useEffect
  - Estados: loading, error, store (datos globales)
  - Funciones CRUD que actualizan el store local (optimistic UI)
  - Routing condicional segun autenticacion:
    - No autenticado: LoginPage
    - Autenticado: Layout con rutas protegidas
  - ProtectedRoute: componente wrapper que verifica rol requerido

Paginas y rutas:
  /                  -> DashboardPage (resumen general)

12. INTEGRACION N8N
================================================================================

Endpoint: /webhook/n8n/* (protegido por X-API-Key header)

La integracion con n8n permite automatizar la busqueda de CVEs desde la NVD
(National Vulnerability Database) cuando se crea una nueva empresa.

Flujo:
  1. Admin crea una empresa via POST /empresas
  2. CompanyService.create_company() llama a trigger_n8n_fetch()
  3. trigger_n8n_fetch() hace POST a config.N8N_WEBHOOK_URL con { "trigger": "new_company" }
  4. n8n recibe el webhook, busca CVEs relacionados con las tecnologias de la empresa
  5. n8n crea vulnerabilidades via POST /webhook/n8n/vulnerabilidades

Webhooks n8n disponibles:
  GET  /webhook/n8n/empresas               -> lista empresas (para que n8n las conozca)
  POST /webhook/n8n/vulnerabilidades       -> crea vulnerabilidad (n8n envia CVEs encontrados)
  GET  /webhook/n8n/vulnerabilidades/existe?cve= -> verifica si un CVE ya existe (evita duplicados)

14. DIAGRAMA DE FLUJO DE AUTENTICACION
================================================================================

LOGIN:
  Cliente                    Servidor
    |                           |
    |-- POST /auth/login ------->|
    |   { username, password }   |
    |                           |-- Verificar password (bcrypt)
    |                           |-- Verificar active=True
    |                           |-- Crear access_token (JWT, 30min)
    |                           |-- Crear refresh_token (JWT + DB, 7dias)
    |<-- 200 OK -----------------|
    |   { access_token,          |
    |     refresh_token, ... }   |
    |                           |
    |-- Almacena tokens en      |
    |   localStorage            |

REQUEST AUTENTICADO:
    |                           |
    |-- GET /vulnerabilidades -->|
    |   Authorization: Bearer   |
    |   <access_token>          |
    |                           |-- Decodificar JWT
    |                           |-- Verificar type="access"
    |                           |-- Cargar usuario de DB
    |                           |-- Verificar active=True
    |                           |-- Verificar token_version coincide
    |<-- 200 OK -----------------|

REFRESH TOKEN (cuando access_token expira):
    |                           |
    |-- POST /auth/refresh ----->|
    |   { refresh_token }       |
    |                           |-- Decodificar JWT refresh
    |                           |-- Buscar en DB (jti, not revoked, not expired)
    |                           |-- Marcar como revoked=True
    |                           |-- Crear NUEVO access_token
    |                           |-- Crear NUEVO refresh_token
    |<-- 200 OK -----------------|
    |   { access_token,          |
    |     refresh_token }        |
    |                           |
    |-- Actualiza localStorage  |

LOGOUT:
    |                           |
    |-- POST /auth/logout ------>|
    |   Authorization: Bearer   |
    |                           |-- Revocar todos los refresh_tokens del user
    |                           |-- Incrementar token_version
    |<-- 200 OK -----------------|
    |                           |
    (cualquier access_token previo queda invalido)
